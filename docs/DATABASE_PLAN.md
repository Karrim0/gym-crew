# Database plan

The first executable schema now lives at:

`supabase/migrations/202607140001_initial_schema.sql`

Apply it to a linked Supabase project with `supabase db push`, then replace
`src/lib/supabase/types.ts` using the generation command in
`supabase/README.md`.

## Sources of truth

The schema deliberately avoids storing the same relationship twice:

- Group membership lives only in `group_members`. `profiles` does not carry a
  second `group_id` column.
- Friday is represented by the group split row whose `workout_type` is
  `rest`.
- A member's zero-to-two extra rest days live only in
  `profiles.additional_rest_days` and act as a schedule overlay. They do not
  erase that day's split exercises.
- `workout_type` is the only rest/workout discriminator on a split day; there
  is no competing `is_rest_day` boolean.

## Main tables

| Table | Responsibility |
| --- | --- |
| `profiles` | User-facing profile and additional personal rest days. |
| `groups` | Workout group and invite code. |
| `group_members` | Membership and owner/admin/member role. Enforces one group per user for the MVP. |
| `exercises` | Shared exercise library and member-created custom exercises. |
| `split_days` | Seven group rows plus optional personal copy rows. |
| `split_exercises` | Ordered exercise configuration for a split day. |
| `workout_sessions` | Session header, stopwatch duration, status, and idempotency key. |
| `workout_exercises` | Exercises actually performed in a session. |
| `workout_sets` | Set-by-set weight and reps. IDs are generated client-side for safe offline upserts. |
| `personal_records` | Current best record per user/exercise/type. |
| `group_activity` | Safe denormalized group feed used by Realtime. |

## Group and personal splits

`split_days.owner_user_id` distinguishes the two versions:

- `NULL`: shared group split; only owner/admin may edit.
- Current user ID: personal copy; only that user may read/write it.

Partial unique indexes guarantee one row per weekday for each effective split.
The database enforces Friday as rest and prevents other split rows from being
changed to `rest`; extra rest days stay in the profile overlay.

## Offline idempotency

Sessions, workout exercises, and sets use UUID primary keys generated by the
client. Retrying an offline mutation can therefore use an upsert without
creating duplicates. `workout_sessions.client_id` is an additional
idempotency key for batching and conflict detection.

The Dexie layer is normalized: session rows do not embed exercise arrays and
exercise rows do not embed set arrays. The nested domain model is assembled
when data is read for the UI.

## Security

RLS is enabled for every user-facing table. The migration includes helper
functions and initial policies for:

- reading profiles and activity only inside the current group;
- writing only the authenticated user's workouts and personal split;
- editing the group split only as owner/admin;
- creating/joining groups through `security definer` RPCs rather than raw
  client inserts;
- keeping raw workout data private while exposing a safe activity feed.
